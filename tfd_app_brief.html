TFD App Brief & Technical Specification

Patient Engagement & Practice Management MVP for Thomas Family Dentistry

Executive Summary

What We're Building

A mobile-first web application for managing appointments, patient engagement, and substitute hygienist coverage for a small dental practice (4 employees).

Why It Matters

  • Hygienist coverage crisis: When the hygienist is out, patients are rescheduled manually. Loss of $300-400/day revenue. Need automated substitute network.
  • Scheduling bottleneck: Wife/ops manager can't always answer calls. Need patient self-service booking & rescheduling.
  • Patient leakage: Patients forget appointments (15-25% no-show rate). No touch points between visits. Need automated engagement.
  • Lost consumables revenue: Patients buy whitening kits from Amazon instead of the practice. Need in-app marketplace.

The Solution

An integrated platform that combines:

  • Patient database & appointment management
  • Automated SMS/email reminders
  • Patient portal for self-service booking
  • Substitute hygienist network & alert system
  • Consumables e-commerce (Phase 2)
  • Analytics dashboard (Phase 4)

User Personas & Workflows

Persona 1: Practice Owner/Dentist

Goals: Keep practice running smoothly, maximize revenue, maintain patient satisfaction
Pain Points: Inconsistent operations, coverage gaps, patient no-shows
Key Workflows:
  • View daily schedule in morning (see if anyone is out)
  • Get alert if coverage is needed (substitute booked automatically or manual confirmation needed)
  • Check revenue metrics (how much did we make? how many no-shows?)
  • Review patient feedback/satisfaction

Persona 2: Operations Manager (Wife)

Goals: Manage day-to-day chaos, reduce phone time, simplify scheduling
Pain Points: Always answering calls, manual rescheduling, finding coverage takes hours
Key Workflows:
  • Patients call/text to book → system shows availability → patient self-books (less phone time)
  • Hygienist calls out sick → system alerts substitute network → coverage confirmed (vs. her making calls)
  • Patients need to reschedule → system sends them rescheduling link → they self-serve (vs. phone call)
  • Follow up on no-shows or missed engagement (app tells her who needs a nudge)

Persona 3: Dental Hygienist

Goals: Do hygiene exams, be comfortable, know schedule in advance
Pain Points: Inconsistent schedule, unclear when she's needed for coverage calls
Key Workflows:
  • Check daily schedule (who's booked? any special notes?)
  • See patient notes before appointments (pre-op information, treatment notes from last visit)
  • Mark tasks (need to send retainer reminder to patient X)
  • Access substitute hygienist network (optional shift picking if she wants extra hours)

Persona 4: Patient

Goals: Easy booking, helpful reminders, convenient care products
Pain Points: Can't reach the office, forget appointments, buy care products elsewhere
Key Workflows:
  • Receive SMS/email reminder before appointment
  • Quick reschedule via link if needed (no phone call)
  • Get post-visit follow-up ("how's your teeth feeling?")
  • Receive engagement messages (time for 6-month cleaning, whitening offer, etc.)
  • Buy consumables (whitening, retainers, care products) through patient portal

Data Models & Database Schema

Core Tables

patients
id (UUID), phone (E.164), email, name, dob, created_at, updated_at, treatment_notes (TEXT), allergies (TEXT), insurance_info (JSON), is_active (boolean)
appointments
id (UUID), patient_id (FK), appointment_type (enum: cleaning, exam, procedure), scheduled_at (datetime), assigned_to (dentist or hygienist), status (enum: scheduled, completed, no_show, cancelled, rescheduled), notes (TEXT), created_at, updated_at
staff
id (UUID), name, role (enum: dentist, hygienist, assistant, admin), phone, email, is_active, created_at
messages (for tracking engagement)
id (UUID), patient_id (FK), message_type (enum: reminder, follow_up, recall, promotional), channel (enum: sms, email), content (TEXT), sent_at, opened_at, clicked_at, created_at
products (for consumables shop)
id (UUID), name, category (enum: whitening, retainers, care_products), price (decimal), description (TEXT), image_url, is_active, created_at
orders (e-commerce)
id (UUID), patient_id (FK), product_id (FK), quantity, total_price, status (enum: pending, paid, ready_for_pickup, shipped), created_at, updated_at
substitute_hygienists
id (UUID), name, phone, email, is_active, is_verified, notes (TEXT), created_at
coverage_alerts (call-out tracking)
id (UUID), staff_member_id (FK), call_out_reason (enum: sick, emergency, other), alert_sent_at, coverage_found (boolean), substitute_id (FK, nullable), created_at

API Endpoints & Integrations

Authentication

POST /auth/login
Request: { phone, code_or_password }
Response: { token, user_type, practice_id }
→ SMS-based login for patients, password login for staff

Appointments (Phase 1)

GET /api/appointments
Query: ?date=2025-03-10&staff_id=xyz
Response: Array of appointments for that day
POST /api/appointments
Request: { patient_id, scheduled_at, type, assigned_to }
Response: Created appointment object. Triggers SMS reminder queue.
PATCH /api/appointments/:id
Request: { status, notes, assigned_to }
Response: Updated appointment

Patient Portal (Phase 1)

GET /api/patients/:id/appointments
Response: List of patient's upcoming appointments
POST /api/patients/:id/reschedule-request
Request: { appointment_id, preferred_times: [] }
Response: Confirmation. Triggers ops notification.

Messaging (Phase 2)

POST /api/messages/send
Request: { patient_id, type, channel, template_id }
Response: Message ID. Triggers SMS/email queue.
GET /api/messages/engagement
Query: ?metric=open_rate&date_range=30d
Response: Engagement metrics

E-Commerce (Phase 2)

GET /api/products
Response: List of available consumables
POST /api/orders
Request: { patient_id, items: [{ product_id, quantity }] }
Response: Order object with Stripe payment link
POST /api/webhook/stripe
Stripe sends payment confirmation → Update order status to "paid"

Coverage Network (Phase 3)

POST /api/coverage-alert
Request: { staff_member_id, reason }
Response: Alert sent to substitute network. Returns confirmation.
POST /api/substitute/accept-shift
Request: { alert_id, substitute_id }
Response: Shift confirmed. Patients notified of substitute. Appointments updated.

External Integrations

SMS (Twilio)

Send reminders, follow-ups, coverage alerts, reschedule confirmations.
Queue: Bull or similar to throttle/batch messages.
Template system: Ops can customize messages without code.

Email (SendGrid or AWS SES)

Backup for patients who don't have SMS enabled.
Same templates as SMS.

Payments (Stripe)

Consumables checkout.
Webhook for payment confirmations.
Fee: 2.2% + $0.30 per transaction (practice absorbs or passes to patient).

Authentication (Optional: Auth0 or Firebase)

For staff logins (dentist, hygienist, ops).
Patients use phone-based SMS login (simpler).

Technical Stack Recommendations

Backend

Framework: Node.js (Express) or Python (FastAPI/Django)
Database: PostgreSQL (relational, reliable, open-source)
Message Queue: Bull (Redis-backed) for SMS/email scheduling
Cache: Redis for session management
Authentication: JWT tokens + SMS-based OTP for patients

Frontend

Framework: React or Vue (mobile-first)
Mobile: PWA (Progressive Web App) to feel native
UI: Tailwind CSS or Material-UI for quick iteration
State: React Context or Vuex (simple, no Redux overhead needed)

Infrastructure

Hosting: Vercel (frontend) + Railway/Render (backend)
Database: Managed PostgreSQL (Supabase, Railway, or AWS RDS)
Object Storage: S3 or Cloudinary for product images
Monitoring: Sentry for error tracking

Why These Choices

  • PostgreSQL: Reliable, scalable, good for relational data (patients, appointments, orders).
  • Node.js + Express: Fast to prototype, great for real-time features (appointment alerts, coverage matching).
  • React: Large ecosystem, component-heavy (perfect for a practice management dashboard).
  • Twilio/SendGrid: Proven, reliable, reasonable pricing. SMS reminders are mission-critical.
  • Stripe: Simple integration, PCI compliant, trusted by patients.
  • Vercel: Auto-deploys on git push, built for React, free tier for testing.

Security & Compliance Considerations

HIPAA (if handling patient health info)

  • Encrypt data in transit (HTTPS everywhere)
  • Encrypt sensitive fields at rest (patient notes, health history)
  • Audit logging (who accessed what, when)
  • Regular backups
  • Data retention policies (comply with state law)

PCI-DSS (for payments)

  • Use Stripe (PCI compliant), never store card data
  • HTTPS for all transactions
  • Regular security audits

General Security

  • Rate limiting on login endpoints (prevent brute force)
  • CSRF tokens for form submissions
  • Input validation / SQL injection prevention
  • Regular dependency updates
  • Monitoring & alerting (Sentry)

Edge Cases & Gotchas to Handle

  1. Time zones: Store all times in UTC, convert for display. Reminders should respect patient's local time.
  2. Double-booking: Prevent dentist/hygienist from being assigned to 2 appointments at same time.
  3. SMS delivery failures: If Twilio fails to deliver, retry queue + fallback to email.
  4. Patient reschedule conflicts: If patient reschedules to a time that's now booked, show error + suggest alternatives.
  5. Coverage alert timing: If call-out happens last-minute (1h before), auto-reschedule patients immediately (don't wait for substitute).
  6. Payment failures: If Stripe charge fails, order stays in "pending" state. Ops gets notification to follow up with patient.
  7. Staff with multiple roles: Dentist might also do hygiene sometimes. Allow flexibility in scheduling.
  8. Patient phone number changes: If patient updates phone, old SMS reminders might bounce. Graceful error handling.
