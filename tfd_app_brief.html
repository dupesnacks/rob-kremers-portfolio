TFD App Brief & Technical Specification

Single App, Role-Based Access | Patient Portal + Practice Dashboard

🔑 Critical Architecture Decision

This is ONE app, NOT two.

Patients log in → See patient portal (appointments, reminders, consumables shop)

Staff log in → See practice dashboard (scheduling, coverage alerts, analytics, campaigns)

Same backend, same database, shared APIs. Authentication + role-based UI rendering determines what users see.

Executive Summary

What We're Building

A single mobile-first web application (PWA) that serves TWO user experiences based on login role:

  • Patient Experience: View appointments, get reminders, reschedule, buy consumables, track orders
  • Staff Experience: Manage scheduling, coverage alerts, patient engagement, e-commerce, analytics

Why This Approach

  • Simpler development: One codebase, one database, one set of APIs
  • Better data consistency: No syncing issues between two apps
  • Easier maintenance: Bug fixes, feature updates apply to both experiences
  • Faster to market: Build incrementally, not in parallel
  • Better UX: Patients see what they need. Staff see what they need. No confusion.

Role-Based Access Model

Users authenticate once. Auth token includes their role. Frontend renders different UI based on role.

Patient Role

Patient

Auth: Phone number + SMS OTP (no password needed)

Visible Features:

  • My Appointments (view, reschedule)
  • SMS/Email Reminders (receive automatically)
  • Engagement Messages (follow-ups, recalls, promotions)
  • Consumables Shop (browse, buy, track orders)
  • My Profile (name, phone, email, preferences)

NOT visible: Staff scheduling, coverage alerts, analytics, campaigns, other patient data

Staff Roles

Practice Admin / Dentist (Owner)

Auth: Email + password (or OAuth)

Visible Features:

  • Dashboard (key metrics, no-show rate, revenue, consumables sales)
  • Schedule (view, manage, assign staff)
  • Coverage Alerts (when someone calls out)
  • Campaigns & Engagement (create reminders, follow-ups, promotions)
  • E-Commerce (add products, set prices, view orders)
  • Staff Management (add/remove staff, permissions)
  • Substitute Network (manage backup hygienists)

Operations Manager (Your Wife)

Auth: Email + password

Visible Features:

  • Schedule (view, manage, assign)
  • Coverage Alerts (log call-outs, find substitutes, reschedule)
  • Patient Reschedule Requests (approve, suggest new times)
  • Reminders & Follow-ups (configure, send, track opens)
  • Orders (view, mark as ready/shipped)
  • Daily Dashboard (today's appointments, alerts, tasks)

NOT visible: Analytics, financial reports, staff management, substitute management

Hygienist

Auth: Email + password

Visible Features:

  • My Schedule (view assigned appointments)
  • Patient Notes (see pre-op info, treatment history)
  • Task List (patient follow-ups, retainer reminders)
  • Optional: Coverage Shifts (see available substitute shifts, accept if interested in extra hours)

NOT visible: Other staff schedules, financial data, e-commerce, analytics

Substitute Hygienist (External)

Auth: Phone number + SMS OTP (similar to patients, but limited)

Visible Features:

  • Coverage Alerts (SMS notification when TFD needs coverage)
  • Shift Details (date, time, patient list, special notes)
  • Shift Acceptance (accept or decline the shift)
  • Patient List (after acceptance, see who's booked)

NOT visible: Financial data, other staff info, past shifts (only current/upcoming)

Technical Architecture (Single App)

Backend (Shared)

Framework: Node.js + Express
Database: PostgreSQL (single database for all roles)
Auth: JWT tokens + role claims
APIs: RESTful (same API serves both patient & staff)
Message Queue: Bull/Redis for SMS/email scheduling

Frontend (One Codebase, Different UIs)

Framework: React or Vue
Architecture: Role-based conditional rendering
- Login screen (determine role)
- Auth context stores role + token
- Components check role: {user.role === 'patient' && <PatientUI />}
- Shared components: Navigation, buttons, modals
- Role-specific pages: Separate PatientDashboard vs StaffDashboard
Mobile: PWA (single install, works for all roles)

Database (Single Schema)

Tables:
- users (id, phone/email, role, password_hash, created_at)
- patients (id, user_id, name, dob, insurance, treatment_notes)
- staff (id, user_id, name, role, phone, is_active)
- appointments (id, patient_id, staff_id, scheduled_at, status)
- messages (id, patient_id, type, channel, content, sent_at, opened_at)
- products (id, name, price, category, is_active)
- orders (id, patient_id, product_id, qty, status)
- substitute_hygienists (id, name, phone, is_verified)
- coverage_alerts (id, staff_id, reason, alert_sent_at, substitute_id)

APIs (Same for Both Roles)

POST /auth/login → {user, token, role}
GET /api/appointments → Role-based filtering (patient sees own, staff sees all)
GET /api/patients/:id → Permission check (patient sees self, staff sees all)
POST /api/messages/send → Role-based (patient can't send, staff can)
GET /api/dashboard → Role-based dashboard data (patient dashboard ≠ staff dashboard)

What Rork Should Build First: Phase 1 MVP

Week 1: Auth + Database Setup

  • Create user roles (patient, dentist, ops, hygienist, substitute)
  • Phone-based SMS login for patients (Twilio)
  • Email+password login for staff
  • JWT token generation + role claim
  • Database schema (users, patients, staff, appointments)

Week 2: Patient Portal + Staff Dashboard (Core Appointment Features)

  • Patient Portal:
    • Login with phone
    • View my appointments
    • Profile page (name, phone, email)
    • Mobile-first responsive design
  • Staff Dashboard (Ops):
    • Login with email
    • Calendar view of all appointments (day/week)
    • Create new appointment
    • Mark as no-show/cancel
    • Basic metrics (today's count, no-show %)

Week 3: SMS Reminders + Reschedule Workflow

  • SMS reminder sent 24h before appointment (Twilio)
  • Patient clicks reschedule in portal
  • Patient sees available slots, picks new time
  • Ops gets notification to confirm/approve
  • Appointment updated, patient gets confirmation SMS

Testing & Polish

  • Unit tests for auth, appointment CRUD, SMS queue
  • Mobile testing (iPhone + Android)
  • Error handling + logging
  • Deployment (Vercel frontend, Railway backend)

Phase 1 Success Metrics

  • Adoption: 80%+ of staff using platform daily by week 3
  • Patient Portal Usage: 50%+ of patients have logged in
  • SMS Delivery: 100+ reminders sent per week, 95%+ delivery rate
  • Self-Service Rescheduling: 30%+ of reschedules happen via portal (vs. phone)
  • No-Show Reduction: 20% reduction in no-shows from baseline (measure week-over-week)
