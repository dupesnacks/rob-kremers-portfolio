Executive Summary & Verdict

✅ YES, THIS IS FEASIBLE AND WORTH BUILDING

Core Insight: You're not building a dental practice management system. You're building a patient engagement & operational efficiency layer that sits on TOP of existing systems like Open Dental, Dentrix, and Curve.

Why It Works: Dentists already spend $200-500/month on practice management software. They're also buying separate tools for patient communication ($100-300/month), patient engagement (if at all), and reminders. Your app consolidates these into ONE simple interface, reduces monthly tech spend, and solves real operational pain points.

Timeline: MVP in 8-12 weeks. Real business viability proof in 6 months (TFD + 5-10 other practices).

Exit Path: $3-10M acquisition by Henry Schein, Philips, or Align within 18-24 months once you have 50-100 practices with clear ROI metrics.

The Play (In One Paragraph)

Independent dentists are losing patients to DSO chains because they can't compete on operational smoothness. They can't book appointments online. They have hygienist coverage crises. Patients forget appointments. They're drowning in phone calls. Your app solves this: patient-facing (easy online booking, SMS reminders, easy rescheduling, follow-up engagement, consumables shop) + practice-facing (call alerts, staff dashboards, coverage matching, engagement automation). You're not replacing Dentrix or Open Dental. You're making them 10x better for practices that can't afford the $10K+ setup and ongoing complexity. You layer on top, pull data via API where possible, build direct patient features around the edges. You own the patient experience. You own the consumables distribution. You own the moat.

The API Landscape (Can We Actually Integrate?)

Open Dental: YES (Best Option)

Open Dental has a well-documented REST API. You can:

  • ✅ Fetch appointment data (scheduled, completed, no-shows)
  • ✅ Fetch patient data (name, phone, email)
  • ✅ Create/update appointments
  • ✅ Get procedure codes, notes
  • ⚠️ Limitations: No direct access to clinical charting. Limited workflow customization.

Cost for integration: ~40-60 hours of dev work to build a solid API layer.

Curve Dental: PARTIAL (Cloud-Based)

Curve has APIs emerging. Less documented than Open Dental, but possible:

  • ✅ Scheduling data accessible
  • ✅ Patient contact info accessible
  • ⚠️ Newer platform, API still maturing

Cost for integration: ~60-80 hours (less documented = more discovery).

Dentrix: NO (Closed System)

Dentrix is proprietary and doesn't offer open APIs for third-party integrations. You would:

  • ⚠️ Have to sync data manually or through workarounds
  • ⚠️ Use middleware (expensive, fragile)
  • ❌ Not recommended as priority

Verdict: Dentrix users are locked in. Skip them for now. Focus on Open Dental + Curve first.

What About the Others?

Weave, SolutionReach, RevenueWell = patient communication tools (NOT practice management). You'll integrate with them by: (1) pulling patient data from PMS, (2) sending SMS/email through your own Twilio/SendGrid, (3) letting practices choose whether to keep their existing tool or use yours.

🎯 Integration Strategy

Phase 1 MVP: Deep integration with Open Dental API. Assume patient & appointment data flows from Open Dental → Your app.

Phase 2: Add Curve Dental support.

Phase 3: "Bring your own data" mode for practices on Dentrix or other closed systems (manual import or lite API via middleware).

All phases: You own patient communication entirely (SMS via Twilio, email via SendGrid). Practices don't need Weave if they use you.

Technology Approach & Integration Strategy

You're NOT Building a Practice Management System

You're building a Patient + Practice Operations Layer that lives on top of Open Dental / Curve / others.

Architecture Philosophy

Do: Build patient-facing features (app, SMS, booking links, consumables shop, follow-up content)

Do: Build practice dashboards (coverage alerts, appointment summaries, engagement metrics)

Do: Integrate with existing PMS via API to read appointment + patient data

Don't: Replicate appointment scheduling, clinical charting, insurance claims

Don't: Try to replace the PMS (you can't and shouldn't)

Example Data Flow: 1. Practice uses Open Dental for core operations 2. Open Dental API syncs appointments + patients to your DB (webhook-driven) 3. Your app: - Sends SMS reminders 24h before - Gives patient link to reschedule - Tracks no-shows, engagement - Shows practice a dashboard - Manages patient consumables shop 4. If coverage needed, alerts substitute network 5. Everything feeds back to Open Dental via API (reschedules, notes, etc.)

Tech Stack (iOS App)

Frontend (Patient + Staff)

  • Native iOS (SwiftUI)
  • MVVM architecture with role-based nav
  • KeyChain for secure token storage
  • Core Data for offline appointment caching
  • APNs (push notifications) for alerts

Backend API

  • Node.js + Express (RESTful)
  • PostgreSQL database
  • JWT authentication with refresh tokens
  • Open Dental API client integration
  • Twilio (SMS sending + OTP verification)
  • SendGrid (email backup)
  • Stripe (payment processing for consumables)
  • Bull/Redis (job queue for scheduled messages)

Key Implementation Details

  • APNs Setup: Practice gets alerts (coverage needed, rescheduled patient, etc.) via push
  • SMS Reminders: Scheduled via Bull job queue. 24h before appointment: "Your cleaning on [date] at [time]. Click to reschedule: [link]"
  • Open Dental Sync: Polling every 30min or webhook-driven when appointment created/modified
  • Patient Data Security: Encrypt sensitive fields (treatment notes). Audit logging for compliance.
  • Offline Mode: iOS app caches appointments locally. Full sync when online.

MVP Scope: Phase 1, 2, 3 (3 Months Total)

Phase 1: Foundation (Weeks 1-4) - "Make The Wife's Life Easier"

Core Features
  • Patient Mobile App: Login (phone + SMS code), view appointments, reschedule button
  • Staff Dashboard: Today's schedule view, alerts, quick metrics
  • Open Dental Integration: API connection to pull appointments + patients
  • SMS Reminders: Automated 24h before appointment, "Click to reschedule"
  • Reschedule Workflow: Patient clicks link → sees available slots → picks new time → staff notified
  • No-Show Tracking: Staff marks appointment as no-show, system tracks rate

Success Metric: TFD using app, 50%+ of patients have app, 30%+ self-reschedule, 15% reduction in no-shows

Effort: ~80 hours (backend + iOS + integration)

Phase 2: Engagement & Commerce (Weeks 5-8) - "Keep Patients + Capture Revenue"
  • Follow-Up Messages: Auto-send post-appointment ("How did your cleaning feel?", treatment education)
  • Recall Automation: Auto-trigger "Time for 6-month cleaning" based on appointment type + interval
  • Consumables Shop: In-app storefront with whitening kits, retainers, care products
  • Patient Portal Enhancement: View past appointments, visit summaries, educational content
  • Engagement Analytics: Dashboard showing message open rates, product interest, retention metrics

Success Metric: $200-300/month consumables revenue, 40%+ engagement message open rate, 15% of patients browse shop

Effort: ~70 hours (templates, e-commerce, analytics)

Phase 3: Operational Resilience (Weeks 9-12) - "Coverage Never Breaks"
  • Coverage Alert System: Staff logs call-out → system alerts pre-vetted substitute network via SMS
  • Substitute Acceptance: First to accept gets shift, appointment list auto-generated
  • Auto-Reschedule on Gap: If no coverage found in 2 hours, auto-reschedule affected patients + notify them
  • Practice Resilience Score: Dashboard shows "X% coverage success rate this month"

Success Metric: 100% coverage resolution, no lost revenue from call-outs, <2 hour average resolution time

Effort: ~60 hours (alerting system, matching logic)

Why 3 Months is Realistic

  • You're not building from scratch. You're integrating existing services (Twilio, Stripe, Open Dental).
  • You're building ONE app experience (iOS only), not web + iOS + Android.
  • You have real customer (TFD) to iterate with immediately.
  • You're layering ON TOP of existing PMS, not replacing it.
  • Rork has AWS/backend experience. You know the codebase by the end of Phase 1.

Interview Framework for TFD Meeting

Goal: Understand the REAL pain points + current workflows + what TFD would actually use. Come away with concrete data to validate your MVP scope.

Pre-Interview Prep

  • Schedule 60-90 minutes with your wife + dentist + office manager (all three, if possible)
  • Prepare 3-4 devices to share screen or show prototypes
  • Record (with permission) for later review
  • Frame as: "I'm building something to help your practice. I need your honest feedback."

Interview Script (Structured)

Section 1: Current Pain (15 min)

Goal: Validate the pain points we've identified.

Questions:

  • "Walk me through a typical day when things go wrong. What frustrates you most about scheduling / patient communication?"
  • "How many times per week do you get calls from patients wanting to reschedule? How do you handle it now?"
  • "Tell me about the last time [hygienist] called out sick. What happened? How much revenue did you lose? How much time did it take to fix?"
  • "What percentage of patients no-show? How do you remind them?"
  • "Do patients buy whitening kits or retainers? Where? How much revenue are you losing to Amazon / mail-order?"
  • "If you could fix ONE thing about running your practice, what would it be?"

What You're Looking For: Specific numbers (no-show %, coverage incidents/year, consumables lost), emotional reactions (frustration with current processes), willingness to try something new.

Section 2: Current Software Stack (10 min)

Goal: Understand what they use + pain points with current tools.

Questions:

  • "What practice management software do you use? How long? Happy with it?"
  • "Do you use Weave, SolutionReach, or anything for patient reminders? Cost? Do you like it?"
  • "What other software do you pay for monthly?" (imaging, scheduling, portal, etc.)
  • "If you could design your IDEAL software to replace everything, what would it look like?"
  • "Have you heard of [Curve, CareStack, etc.]? What do you think?"

What You're Looking For: Software frustrations, price sensitivity, openness to replacing tools, integration headaches they're experiencing.

Section 3: Validate MVP Scope (20 min)

Goal: Show them Phase 1 concepts + get their reaction. Do they WANT this?

Pitch + Feedback:

  • "I'm building an app that does three things: (1) Patients can reschedule appointments with a click, (2) Automatic reminders reduce no-shows, (3) When someone calls out sick, we alert a network of substitute hygienists who can fill in that day. No more scrambling. Would that help you?"
  • Show mockup of patient app: appointments view, reschedule button, SMS reminder preview
  • Show mockup of staff dashboard: today's schedule, coverage alert, metrics
  • "How would you use this? What would change about your day?"
  • "What's missing? What else would you want?"
  • "How much would you pay for this? Would you replace Weave?"

What You're Looking For: "Yes, I would use this." Specific feature requests. Price willingness. Concerns about switching from current tools.

Section 4: Consumables & Revenue (10 min)

Goal: Understand if they'd want a consumables shop + how much revenue they're missing.

Questions:

  • "Do patients ask you about whitening kits or retainer replacements? Where do they end up buying them?"
  • "If we built a shop IN the app and handled fulfillment, would you sell whitening kits / retainers through it?"
  • "What would you want to sell? Whitening? Retainers? Toothbrushes? Care products?"
  • "How much do you think you're losing to Amazon right now? Per month?"
  • "Would you want us to handle it (affiliate or wholesale), or would you want to manage inventory yourself?"

What You're Looking For: Interest level in consumables. Current lost revenue estimate. Preference for hands-off (affiliate) vs. wholesale. Product mix they'd prioritize.

Section 5: Willingness to Be a Beta User (10 min)

Goal: Commitment to using the MVP + providing feedback.

Questions:

  • "If I built this and launched it in 3 months, would you be willing to be our first real user?"
  • "What would you need from us to make it a no-brainer to switch?"
  • "How often would you give me feedback? Weekly? As issues come up?"
  • "Would your team (staff, dentist) be open to using a new app?"
  • "What's the deal-breaker that would make you NOT use this?"

What You're Looking For: Clear yes/no on beta participation. Specific requirements. Team buy-in likelihood. Potential deal-breakers.

Post-Interview Actions

  • Summarize: "So if I heard you right, the biggest pain is X. You'd use Y if it did Z. You'd be willing to pay $A. Is that right?"
  • Get their permission to contact them when MVP is ready (4-8 weeks)
  • Send follow-up email with notes + next steps
  • Adjust MVP scope based on their feedback

Interview Success Criteria

You leave the meeting with:

  • ✅ Quantified pain points (e.g., "We lose $400-600/week to no-shows and coverage gaps")
  • ✅ Clear MVP scope feedback (e.g., "Reminders are nice but coverage matching is CRITICAL")
  • ✅ Pricing comfort level (e.g., "I'd pay $250-350/mo if it replaced Weave")
  • ✅ Beta participation commitment (e.g., "Yes, I'll use it. Launch in 3 months?")
  • ✅ Team buy-in assessment (e.g., "Staff would use this immediately")
  • ✅ Feature prioritization (e.g., "Do coverage matching FIRST, consumables later")
