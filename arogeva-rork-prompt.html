🎯 The Core Concept

Unlike calorie-counting apps, Arogeva answers:

  • "What should I eat if I have PCOS + gut issues?" → Condition-specific protocols from Mitali
  • "Is this restaurant meal safe for my condition?" → Smart restaurant menu filtering via API
  • "What should I shop for?" → Personalized grocery lists tied to protocols
  • "What do my lab values mean?" → Functional medicine interpretation linked to food choices
  • "How do my conditions connect?" → Shows interactions (e.g., gut health impacts hormones)

⚡ Core Features to Build

HIGH PRIORITY 1. Condition Interconnection Logic

Why: Functional medicine recognizes conditions interact. PCOS + Gut Health needs different guidance than each alone.

Implementation:
• Create a Condition relation graph (define which conditions relate to which)
• When user selects 2+ conditions, show how they interact
• Router shows combined protocol, not separate ones
Example: User picks Gut Health + Brain & Focus → App shows: "Leaky gut impacts cognition. This protocol addresses both."
Data Structure: Each Condition has array of related_conditions and a combined_protocol_id

HIGH PRIORITY 2. Personalized Protocol System

Why: This is Mitali's IP. Needs to scale without losing her voice.

Implementation:
• Admin panel (Mitali-only): Create/edit protocols with steps, duration, recipes
• Protocol Schema: name, description, phases (Phase 1: Elimination, Phase 2: Reintro, etc.), duration_days, recipes[], supplements[], foods_to_avoid[], clinician_notes
• Smart assignment: Based on user profile (age, diet type, conditions, activity), recommend 1 primary protocol
• User can view: Protocol timeline, phase-specific recipes, what to expect
Example Protocol:
  Name: "Gut Reset for Inflammation"
  Duration: 30 days
  Phase 1 (Days 1–7): Khichdi fast + bone broth
  Phase 2 (Days 8–21): Introduce cooked veggies, ghee
  Phase 3 (Days 22–30): Add proteins, legumes gradually

HIGH PRIORITY 3. Regional Recipe Library (India-First)

Why: India/GCC version needs 180+ regional recipes curated by Mitali.

Implementation:
• Recipe Schema: name, cuisine (South Indian, North Indian, Bengali, etc.), ingredients, method, nutrition_facts, condition_tags[], prep_time_min, difficulty
• Smart Filtering: By condition + diet type + cuisine preference + prep time + dietary restrictions
• Regional Variants: Store recipe base + variants (e.g., "Khichdi" has Gujarati, Tamil, Bengali versions)
• Integration: Recipes appear in protocols, can be favorited, searchable
Content Requirements:
  — 180 base recipes across 8 Indian cuisines
  — Each tagged with 1–3 condition categories
  — Include nutrition breakdown (macros, key micronutrients)
  — Video links optional (YouTube embeds for demo)
Data Input: CSV or admin panel for Mitali to add/edit recipes

MEDIUM PRIORITY 4. Travel Meal Navigation (API Integration)

Why: Users eating out need real-time guidance. Zomato (India) + Uber Eats/GrubHub (US) integration.

Implementation:
• Feature: "Eat Out Safely" tab
• User enters: Current location + restaurant name (or picks from map)
• App fetches menu via API + shows:
  — "Safe Meals" (green): Align with their condition
  — "Modify These" (yellow): Safe with adjustments
  — "Avoid" (red): Conflicting with condition
• Tap meal → see nutrition facts + condition-specific notes ("High in lectins—avoid if autoimmune")
APIs Required:
  — Zomato API (India): Restaurant menu data
  — Google Places API: Geolocation + restaurant discovery
  — Nutritionix or USDA API: Nutrition data fallback
Phase 2 Launch: Start with Zomato; add US APIs in Phase 3

MEDIUM PRIORITY 5. Meal Logging & Condition Scoring

Why: Engagement driver. Users need to see progress toward better condition scores.

Implementation:
• Daily Dashboard: Shows today's score, trends, quick actions
• Logging: User can log meals (free-text or photo upload) + symptoms (bloating? energy? skin?)
• Scoring Algorithm:
  — Score = 0–15 per condition
  — Based on: Meals logged (alignment %), symptoms reported, protocol compliance
  — Update daily; show trend graph (last 30 days)
• Motivation: "Your Gut Health improved 3 points this week! Keep going."
Tech: Store meal_logs + symptom_logs with timestamps; compute scores daily

MEDIUM PRIORITY 6. Lab Value Interpretation

Why: Clinical credibility. Users upload bloodwork → app explains what it means for their conditions.

Implementation:
• Feature: "My Labs" section
• User uploads: Lab report (PDF/photo) or enters values manually
• App recognizes key markers: Hemoglobin, ferritin, glucose, HbA1C, TSH, hormone panels, gut markers (zonulin, etc.)
• Shows Interpretation:
  — "Your ferritin is low (12 ng/mL). This is common with your Anemia. Follow the Anemia protocol recipe list—prioritize [iron-rich foods]."
• Safe guardrail: Always include "Discuss with your doctor" + no direct diagnosis
Data Setup: Define 10–15 key lab markers; map each to condition-specific advice + recipe recommendations
Phase 2 Feature: Can add this after MVP if needed

LOW PRIORITY 7. Notifications & Daily Lessons

Why: Re-engagement + education. Users get daily nudges + micro-learning.

Implementation:

• Daily Lesson: Pop-up or tab showing Mitali's clinical insight (e.g., "Why fermented foods heal the gut")
• Could be: Text + optional YouTube video link
Content Management: Admin panel for Mitali to author lessons
Phase 1: Manual WhatsApp messages; Phase 2: Automate via Twilio/WhatsApp Business API

LOW PRIORITY 8. Multi-Language Support

Why: India market includes Hindi, Tamil, Telugu speakers. Lower-literacy users benefit from audio.

Implementation:

⚠️ Important Constraints & Notes for Rork AI

  • Founder as Authority: Don't bury Mitali. Include her name, credentials (MS, RD, CLT), photo on every major screen. All content should feel like it's from her or clearly AI-assisted.
  • Clinical Accuracy: Never diagnose. Always say "Discuss with your doctor" when interpreting labs or symptoms. Frame as educational guidance, not medical advice.
  • Data Privacy is a Feature: Explicitly state "Your data is encrypted and never sold" on relevant screens. Build trust through transparency.
  • Regional Customization: Build architecture to swap recipes/protocols by region (India vs. US vs. GCC). Don't hardcode region-specific content.
  • Scalability for Conditions: When adding new conditions, app should automatically adjust protocols + recipe recommendations. Use relational logic, not hardcoded rules.
  • Color Accessibility: Dark teal background with white text passes WCAG. Test with colorblind mode (protanopia, deuteranopia).
  • Performance: Recipe library might be large (180+). Use pagination or lazy loading; don't load all recipes at once.
