### 2. LLM Constraint Rules Defined
Created 10 hard validation rules to prevent invalid suggestions:
- **Rule 1**: Never suggest liquid foods (except beverages) as primary foods
- **Rule 2**: Avoid bitter flavors for selective eaters (blocked: dark leafy greens, coffee, strong spices)
- **Rule 3**: Don't combine 3+ bold flavors (e.g., salty + sour + spicy)
- **Rule 4**: Temperature + Texture combos validated (e.g., hot soup = acceptable; cold crunchy = acceptable; hot crunchy = risky)
- **Rule 5**: Only suggest foods in approved 100+ database
- **Rule 6**: Color variety tracking prevents monochrome diets
- **Rule 7**: Food group diversity required (don't suggest only proteins)
- **Rule 8**: Bridge foods must have ≥1 sensory attribute in common with already-safe foods
- **Rule 9**: Sensory distance calculation prevents massive jumps (texture diff ≤1 step, flavor ≤1 step)
- **Rule 10**: Weekly batch processing (not real-time) prevents recommendation spam

### 3. Implementation & Cost Strategy
- **Client-side validation**: All guardrails implemented as validation rules, not API calls → stays within budget
- **Threshold**: 3-5 foods logged before bridge food suggestions enabled
- **Cost impact**: Zero additional API overhead (validation is deterministic logic)
- **LLM role**: Classification (attributes) + insights (narratives), not suggestion generation (rule-based instead)
